# pocket-wiki

A personal knowledge base combining Graphify + LLM Wiki patterns.
Converts raw sources into a knowledge graph, then compiles human-readable wiki pages.

## Frontmatter Rules

**Source pages** (`wiki/sources/`):
```yaml
---
title:
type: source
author:
added: YYYY-MM-DD
domain:
source_url:
source_file:
status: summarized | reviewed | superseded
---
```

**Concept pages** (`wiki/<domain>/`):
```yaml
---
title:
type: concept
domain:
tags: []
perspective: []   # optional: [systems, practitioner, theory, history, interview, math]
updated: YYYY-MM-DD
status: draft | stable | archived
---
```

- `[[wikilinks]]` go in the body `## Related` section only ‚Äî never in frontmatter
- `graph/` files are auto-generated by Graphify ‚Äî never modify
- `perspective` records the angle the page was written from (multiple values allowed)

## INGEST

Two modes:
- **Quick mode (default)** ‚Äî `/pocket-wiki <url or title>` runs without discussion. Use sensible defaults and proceed.
- **Discuss mode** ‚Äî `/pocket-wiki discuss <url or title>` triggers the full perspective conversation.

When the user gives a source name or URL without keywords, run quick mode.

### 1. Collect source

If a URL is given:
```
python -m graphify add <url> --dir raw/crawled
```

If only a title/keyword is given: find the most relevant URL on the web and run the command above.
If it's a file: guide the user to copy it into `raw/files/`.

### 2. Update graph

```
Invoke the graphify skill: /graphify <POCKET_ROOT>/raw --update
```

### 3. Discussion (mode-dependent)

**Quick mode (default)** ‚Äî fully autonomous, no user input:
- **Auto-infer `perspective`** from source type and content. Never leave empty. Inference rules:
  - Paper / arxiv / academic source ‚Üí `theory` (add `math` for heavy formal content)
  - Blog / tutorial / how-to / docs with code ‚Üí `practitioner`
  - System internals / architecture / OS / hardware ‚Üí `systems`
  - Tech evolution / version comparison / changelog ‚Üí `history`
  - Interview prep / Q&A format ‚Üí `interview`
  - Combine multiple when content spans angles (e.g. applied ML paper ‚Üí `[theory, practitioner]`)
- If similar pages found (3+ shared tags or near-identical title), **auto-update** existing pages without asking. Mention what was auto-updated in the final summary.
- Do not block on user input. Proceed to Step 4.

**Discuss mode** (when invoked with `discuss` keyword):
- Do NOT write wiki pages yet.
- Before discussing perspective, check if existing pages in the same domain have 3+ overlapping tags or a similar title.
- If found, inform the user: "Similar page already exists: [[X]] (overlapping tags: [...]) ‚Äî options: update existing / create separate page / merge"
- Then share key claims and interesting points, and ask which perspective(s) to emphasize.
- Allowed values: `systems`, `practitioner`, `theory`, `history`, `interview`, `math` (multiple allowed)
- Reflect the user's direction when writing pages.

### 4. Write source page

Create `wiki/sources/<slug>-source.md`:
- Summary of the source's key claims
- Notes reflecting the user's perspective
- frontmatter: type=source
- A single source can touch 10‚Äì15 wiki pages

### 5. Update concept/entity pages

Identify concepts and entities from the source.
- If a page exists ‚Üí update it (explicitly note any contradictions with existing content)
- If not ‚Üí create `wiki/<domain>/<concept>.md`
- frontmatter: type=concept, **perspective** (auto-inferred in quick mode; user-chosen in discuss mode)

**Update strategy:**
- New information that extends existing content ‚Üí append to the relevant section
- Information that contradicts existing content ‚Üí add a `> ‚öÝÔ∏è Contradiction: [source] says X, but [other source] says Y` blockquote in the affected section
- Source attribution ‚Üí add `[[<slug>-source]]` to the `## Related` section of the concept page

### 6. Update _meta/

```
_meta/index.md ‚Äî add new pages (link + one-line summary)
_meta/log.md   ‚Äî append work record
```

log.md format (grep-parseable):
```
## [YYYY-MM-DD] ingest (quick|discuss) | <source name>
pages created/updated: page1, page2, ...
```
Tag the entry with `(quick)` or `(discuss)` so the user can later filter for review.

If Step 3 (discuss mode only) resulted in a **wiki-level structural choice** the user wants to remember (page merge, page split, new domain creation, naming convention), append to `_meta/decisions.md` (see that file's template).

Do NOT log changes to the pocket-wiki tool itself (skill behavior, schema fields, lint rules) ‚Äî those belong in git history, not decisions.md.

## QUERY

When the user asks a question, follow these steps.

### 1. Traverse the graph

```
python -m graphify query "<question>" --budget 2000
```

Identify relevant nodes. This is the navigator ‚Äî it tells you what to read next.

### 2. Read wiki pages or raw files

- wiki exists AND `status: stable` ‚Üí read wiki only (trusted knowledge)
- wiki exists AND `status: draft` ‚Üí read wiki AND the raw source file (cross-check; drafts may have errors)
- wiki doesn't exist ‚Üí read the raw source file

### 3. Answer + ingest if needed

Answer the question. If you read raw files, also write wiki pages (INGEST steps 4‚Äì6).
Next time the same topic comes up, the wiki will be ready.

Valuable answers (comparison tables, analysis, newly discovered connections) ‚Üí save as a new wiki page too.

Log to log.md:
```
## [YYYY-MM-DD] query | <question summary>
answer saved at: wiki/domain/page.md (if saved)
```

## LINT

Run a health check on the wiki when requested.

1. Compare `_meta/index.md` against actual files ‚Äî find missing or mismatched entries
2. Find `[[wikilinks]]` in body text with no corresponding page (outbound orphans)
2.5. **Semantic overlap** ‚Äî find concept page pairs in the same domain sharing 3+ tags (merge/distinguish candidates)
3. Find pages not linked to by any other page (inbound orphans)
4. Find pages with `status: draft` that haven't been updated in a long time
4.5. **Unlinked mentions** ‚Äî find page titles appearing as plain text in other pages without `[[...]]` wrapping
5. Find contradictions between concept pages
6. Identify data gaps ‚Äî suggest questions to investigate and sources to add

Log to log.md:
```
## [YYYY-MM-DD] lint
issues found: ...
suggested next sources: ...
```

If lint leads to a **wiki-level structural action** by the user (merging two pages, splitting a page, renaming, contradiction-resolution), record it in `_meta/decisions.md`. Tool-level changes (new lint rules, schema fields) belong in git history, not decisions.md.

## REVIEW

For inspecting and refining wiki pages ‚Äî assigning perspective, promoting status, fixing content. Works on any wiki page, not just quick-mode pages.

### `/pocket-wiki review` (no argument)

List pages that may need review, in priority order:
1. **Recently quick-ingested pages** (last 7 days) ‚Äî verify auto-inferred perspective
2. **Stable pages** with `> ‚öÝÔ∏è Contradiction:` blockquotes still present
3. **Drafts** older than 30 days that haven't been promoted
4. **Stable pages updated by recent ingest** ‚Äî content drift may have introduced inaccuracies

Show: slug ¬∑ status ¬∑ perspective ¬∑ domain ¬∑ updated ¬∑ last source. Then ask which to review.

### `/pocket-wiki review <argument>`

Argument resolution:
- Matches a page slug exactly ‚Üí interactive review for that page
- Matches a domain folder ‚Üí list reviewable pages in that domain
- Otherwise ‚Üí fuzzy-match against page slugs, offer suggestions

### Interactive review for a single page

1. Read the page and any linked source files for context
2. Show summary: current frontmatter + 2‚Äì3 sentence content recap
3. Ask, in order (skip if not applicable):
   - **Perspective** ‚Äî assign or change `perspective` field
   - **Status** ‚Äî promote: `draft` ‚Üí `stable`, or `stable` ‚Üí `archived`
   - **Tags** ‚Äî add/remove
   - **Content** ‚Äî apply user-dictated edits (resolve contradiction, add section, etc.)
4. Update the page's `updated` field to today
5. Append to `_meta/log.md`:
   ```
   ## [YYYY-MM-DD] review | <slug>
   changes: <what changed>
   ```

If review involves a **structural decision** (new precedent, contradiction-resolution rule), also append to `_meta/decisions.md`.

## DECISIONS

`decisions.md` records **wiki-level** structural choices ‚Äî page splits, merges, domain boundaries, contradiction resolutions, naming conventions. NOT for changes to the pocket-wiki tool (those live in git history).

Invoked via `/pocket-wiki decisions` or `/pocket-wiki decisions add <title>`.

### List (`decisions`)

Read `_meta/decisions.md` and display all wiki decisions in reverse chronological order.
Each entry: date + title + one-line summary.
If the file only has the example template (no real entries yet), say so.
Ask if the user wants to add a new decision or review one in detail.

### Record (`decisions add <title>`)

Guide the user through the entry interactively:
1. **Context** ‚Äî why did this come up while writing the wiki?
2. **Decision** ‚Äî what wiki change was made and how?
3. **Impact** ‚Äî which pages were affected, what links were updated?
4. **Alternatives** ‚Äî options considered but rejected? (omit if none)

Append to `_meta/decisions.md` (newest first, below the marker):
```
## [YYYY-MM-DD]: <title>
- **Context**: ...
- **Decision**: ...
- **Impact**: ...
- **Alternatives**: ... (only if applicable)
```

If the user is trying to log a tool/skill change rather than a wiki change, redirect them: "That's a pocket-wiki tool change ‚Äî git history is the right place. decisions.md is for wiki structure decisions."

## Rules

- Never modify or delete files in `raw/`
- Never modify files in `graphify-out/`
- Never modify files in the `graph/` folder
- Quick mode is the default ‚Äî proceed without blocking the user. Use discuss mode only when the `discuss` keyword is explicitly given.
