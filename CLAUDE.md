# pocket-wiki

A personal knowledge base combining Graphify + LLM Wiki patterns.
Converts raw sources into a knowledge graph, then compiles human-readable wiki pages.

## Structure

```
<root>/
├── raw/                    Source files (never modify)
│   ├── files/              Manually added files
│   └── crawled/            Fetched from the web
├── graphify-out/           Graphify output (never modify)
│   ├── graph.json          Persistent knowledge graph
│   └── GRAPH_REPORT.md
└── LLM Wiki/               Obsidian vault
    ├── graph/              Graphify --obsidian output (never modify)
    ├── wiki/
    │   ├── sources/        Per-source summary pages (<slug>-source.md)
    │   └── <domain>/       Concept/entity pages (<slug>.md)
    └── _meta/
        ├── index.md        Full wiki index (updated on every ingest)
        ├── log.md          Work log (append-only) — what was done
        ├── decisions.md    Structural decision history (ADR) — why things are this way
        └── schema.md       Frontmatter rules
```

## Frontmatter Rules

**Source pages** (`wiki/sources/`):
```yaml
---
title:
type: source
author:
added: YYYY-MM-DD
domain:
source_url:
source_file:
status: summarized | reviewed | superseded
---
```

**Concept pages** (`wiki/<domain>/`):
```yaml
---
title:
type: concept
domain:
tags: []
perspective: []   # optional: [systems, practitioner, theory, history, interview, math]
updated: YYYY-MM-DD
status: draft | stable | archived
---
```

- `[[wikilinks]]` go in the body `## 관련` section only — never in frontmatter
- `graph/` files are auto-generated by Graphify — never modify
- `perspective` records the angle the page was written from (multiple values allowed)

## INGEST

When the user gives a source name or URL, run all steps below.
The user only needs to say: "add X" or "ingest X".

### 1. Collect source

If a URL is given:
```
python -m graphify add <url> --dir raw/crawled
```

If only a title/keyword is given: find the most relevant URL on the web and run the command above.
If it's a file: guide the user to copy it into `raw/files/`.

### 2. Update graph

```
Invoke the graphify skill: /graphify <POCKET_ROOT>/raw --update
```

### 3. Discuss with user

Do NOT write wiki pages yet.

**Before discussing perspective**, check if existing pages in the same domain have 3+ overlapping tags or a similar title.
If found, inform the user: "Similar page already exists: [[X]] (overlapping tags: [...]) — options: update existing / create separate page / merge"

Then share key claims and interesting points, and ask which perspective(s) to emphasize.
Allowed values: `systems`, `practitioner`, `theory`, `history`, `interview`, `math` (multiple allowed)
Reflect the user's direction when writing pages.

### 4. Write source page

Create `wiki/sources/<slug>-source.md`:
- Summary of the source's key claims
- Notes reflecting the user's perspective
- frontmatter: type=source
- A single source can touch 10–15 wiki pages

### 5. Update concept/entity pages

Identify concepts and entities from the source.
- If a page exists → update it (explicitly note any contradictions with existing content)
- If not → create `wiki/<domain>/<concept>.md`
- frontmatter: type=concept, **perspective** (value chosen in Step 3)

**Update strategy:**
- New information that extends existing content → append to the relevant section
- Information that contradicts existing content → add a `> ⚠️ Contradiction: [source] says X, but [other source] says Y` blockquote in the affected section
- Source attribution → add `[[<slug>-source]]` to the `## 관련` section of the concept page

### 6. Update _meta/

```
_meta/index.md — add new pages (link + one-line summary)
_meta/log.md   — append work record
```

log.md format (grep-parseable):
```
## [YYYY-MM-DD] ingest | <source name>
pages created/updated: page1, page2, ...
```

If Step 3 resulted in a **structural decision** (merge, split, new domain, new frontmatter field), also append to `_meta/decisions.md`:
```
## [YYYY-MM-DD]: <decision title>
- **Context**: why this came up
- **Decision**: what was decided and how
- **Impact**: what changes as a result
- **Alternatives**: considered options and why rejected (omit if none)
```

## QUERY

When the user asks a question, follow these steps.

### 1. Traverse the graph

```
python -m graphify query "<question>" --budget 2000
```

Identify relevant nodes. This is the navigator — it tells you what to read next.

### 2. Read wiki pages or raw files

- wiki exists AND `status: stable` → read wiki only (trusted knowledge)
- wiki exists AND `status: draft` → read wiki AND the raw source file (cross-check; drafts may have errors)
- wiki doesn't exist → read the raw source file

### 3. Answer + ingest if needed

Answer the question. If you read raw files, also write wiki pages (INGEST steps 4–6).
Next time the same topic comes up, the wiki will be ready.

Valuable answers (comparison tables, analysis, newly discovered connections) → save as a new wiki page too.

Log to log.md:
```
## [YYYY-MM-DD] query | <question summary>
answer saved at: wiki/domain/page.md (if saved)
```

## LINT

Run a health check on the wiki when requested.

1. Compare `_meta/index.md` against actual files — find missing or mismatched entries
2. Find `[[wikilinks]]` in body text with no corresponding page (outbound orphans)
2.5. **Semantic overlap** — find concept page pairs in the same domain sharing 3+ tags (merge/distinguish candidates)
3. Find pages not linked to by any other page (inbound orphans)
4. Find pages with `status: draft` that haven't been updated in a long time
4.5. **Unlinked mentions** — find page titles appearing as plain text in other pages without `[[...]]` wrapping
5. Find contradictions between concept pages
6. Identify data gaps — suggest questions to investigate and sources to add

Log to log.md:
```
## [YYYY-MM-DD] lint
issues found: ...
suggested next sources: ...
```

If lint results in a **structural change** (schema update, merge/split, new rule), also append to `_meta/decisions.md`.

## DECISIONS

Invoked via `/pocket-wiki decisions` or `/pocket-wiki decisions add <title>`.

### List (`decisions`)

Read `_meta/decisions.md` and display all decisions in reverse chronological order.
Each entry: date + title + one-line summary of what was decided and why.
Ask if the user wants to add a new decision or review one in detail.

### Record (`decisions add <title>`)

Guide the user through the entry interactively:
1. **Context** — why was this decision needed?
2. **Decision** — what was decided and how?
3. **Impact** — what changes in existing pages or workflow?
4. **Alternatives** — options considered but rejected? (omit if none)

Append to `_meta/decisions.md`:
```
## [YYYY-MM-DD]: <title>
- **Context**: ...
- **Decision**: ...
- **Impact**: ...
- **Alternatives**: ... (only if applicable)
```

## Rules

- Never modify or delete files in `raw/`
- Never modify files in `graphify-out/`
- Never modify files in the `graph/` folder
- Never write wiki pages without discussing with the user first (INGEST step 3)
